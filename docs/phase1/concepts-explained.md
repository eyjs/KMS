## 1. 왜 문서 관리가 필요한가?

### 일상에서의 문제

여러분의 스마트폰 사진첩을 생각해보세요.

처음에는 사진이 몇 장 없어서 찾기 쉬웠습니다. 하지만 시간이 지나면서 수천 장이 쌓이면 어떻게 될까요?

- "작년 제주도 여행 사진 어디 있지?"
- "아이 돌잔치 사진 찾아야 하는데..."
- "이 사진이 올해 건가, 작년 건가?"

결국 스마트폰은 **"날짜별 폴더"**와 **"앨범"** 기능을 만들었습니다. 사진에 자동으로 날짜를 붙이고, 우리가 "가족", "여행", "음식" 같은 앨범을 만들 수 있게 했죠.

### 회사에서의 문제

회사의 문서도 마찬가지입니다.

처음에는 문서가 몇 개 없어서 괜찮았습니다. 하지만 수백, 수천 개가 쌓이면:

- "이 시책 자료 어디 있었지?"
- "이게 최신 버전 맞나?"
- "삼성생명 건가, 한화생명 건가?"
- "이 자료 아직 유효한 거야?"

사진첩처럼, 문서에도 **체계적인 정리 방법**이 필요합니다.

### 문서 관리의 목표

우리가 만드는 시스템은 이 질문에 답할 수 있게 합니다:

| 질문 | 답을 주는 기능 |
|------|--------------|
| 이 문서 어디 있어? | **분류체계** |
| 지금 써도 되는 거야? | **라이프사이클** |
| 언제까지 유효해? | **신선도** |
| 관련 자료 뭐가 있어? | **관계** |
| 이게 진짜 맞아? | **SSOT** |

하나씩 자세히 알아보겠습니다.

## 2. 도서관에서 배우는 분류의 원리

### 도서관은 어떻게 책을 정리할까?

도서관에 가면 수만 권의 책이 있지만, 원하는 책을 찾을 수 있습니다. 어떻게 가능할까요?

**듀이 십진분류법**이라는 것을 씁니다:

```
000 총류 (백과사전, 신문 등)
100 철학
200 종교
300 사회과학
400 언어
500 자연과학
600 기술과학
700 예술
800 문학
900 역사
```

예를 들어, "한국 역사" 책을 찾고 싶다면:
- 900 역사 → 950 아시아 역사 → 953 한국 역사

이렇게 **큰 분류에서 작은 분류로** 좁혀갑니다.

### 도서관의 책 vs 회사의 문서

| 도서관 | 회사 |
|--------|------|
| 책 | 문서 |
| 분류 번호 | 분류체계 |
| 서가 위치 | 저장 위치 |
| 대출 가능/불가 | 유효/만료 |
| 관련 도서 | 연관 문서 |

도서관이 책을 정리하는 원리를 회사 문서에 적용하면 됩니다.

### 핵심 원리: 계층 구조

```
도서관:
  역사(900)
    └── 아시아 역사(950)
          └── 한국 역사(953)
                └── 조선시대(953.5)

회사:
  보험사
    └── 삼성생명
          └── 종신보험
                └── 상품설명서
```

**큰 것에서 작은 것으로**, **일반적인 것에서 구체적인 것으로** 내려갑니다.

## 3. 문서의 주소: 분류체계

### 집 주소처럼 생각하기

여러분의 집 주소를 생각해보세요:

```
서울특별시 강남구 테헤란로 123번지 A아파트 101동 1001호
```

이 주소만 있으면 전 세계 어디서든 여러분의 집을 찾을 수 있습니다.

문서도 마찬가지입니다. 문서에 "주소"를 붙여주면, 수천 개 중에서도 정확히 찾을 수 있습니다.

### 문서의 주소 = 분류 경로

보험 GA에서 문서의 "주소"는 이렇게 만듭니다:

```
보험사 > 상품 > 문서유형
```

예시:
```
삼성생명 > 종신보험 > 상품설명서
KB손해보험 > 어린이보험 > 시책
한화생명 > 암보험 > 수수료표
```

### 세 가지 질문 (3-Axis)

모든 문서에 세 가지 질문을 합니다:

| 질문 | 분류 축 | 예시 |
|------|--------|------|
| **누구** 것인가? | 보험사 | 삼성생명, 한화생명, KB손해보험 |
| **무슨** 상품인가? | 상품 | 종신보험, 어린이보험, 암보험 |
| **어떤** 문서인가? | 문서유형 | 시책, 수수료표, 상품설명서, 약관 |

이 세 가지를 조합하면 문서의 "주소"가 됩니다.

### 왜 세 가지인가?

너무 적으면 구분이 안 되고, 너무 많으면 복잡해집니다.

**1가지만 쓴다면:**
```
문서유형: 상품설명서
→ 어느 보험사? 어느 상품? 모름
```

**5가지를 쓴다면:**
```
보험사 > 상품 > 문서유형 > 작성자 > 작성일 > 버전
→ 너무 복잡, 기억하기 어려움
```

**3가지가 적당:**
```
삼성생명 > 종신보험 > 상품설명서
→ 명확하고 기억하기 쉬움
```

### 실제 분류 예시

**삼성생명 종신보험 관련 문서들:**

```
삼성생명
  └── 종신보험
        ├── 상품설명서 ← 상품 개요
        ├── 약관 ← 법적 조건
        ├── 수수료표 ← 설계사 수수료
        ├── 시책 ← 판매 인센티브
        └── 판매스크립트 ← 상담 가이드
```

같은 보험사, 같은 상품이라도 문서유형이 다르면 다른 문서입니다.

## 4. 문서의 생애: 라이프사이클

### 사람처럼 문서도 태어나고 은퇴한다

사람에게 생애 주기가 있듯이, 문서에도 생애 주기가 있습니다.

**사람의 생애:**
```
출생 → 성장 → 활동기 → 은퇴 → 사망
```

**문서의 생애:**
```
작성 → (검토) → 사용 → 만료
```

### 문서의 세 가지 상태

우리 시스템에서 문서는 세 가지 상태 중 하나입니다:

| 상태 | 한글 | 의미 | 색상 |
|------|------|------|------|
| **DRAFT** | 작성 중 | 아직 완성되지 않음 | 회색 |
| **ACTIVE** | 사용 중 | 지금 유효한 문서 | 초록색 |
| **DEPRECATED** | 만료됨 | 더 이상 쓰면 안 됨 | 빨간색 |

### 상태 전환 규칙

```
    ┌─────────┐         ┌─────────┐         ┌────────────┐
    │ DRAFT   │ ──────► │ ACTIVE  │ ──────► │ DEPRECATED │
    │ 작성 중  │         │ 사용 중  │         │ 만료됨     │
    └─────────┘         └─────────┘         └────────────┘
```

**규칙:**
1. 새 문서는 항상 "작성 중"으로 시작
2. 완성되면 "사용 중"으로 변경
3. 새 버전이 나오면 이전 버전은 "만료됨"으로 변경

### 왜 상태 관리가 필요한가?

**상태 관리가 없다면:**

```
[공유폴더]
├── 삼성생명_종신보험_수수료표.pdf
├── 삼성생명_종신보험_수수료표_최종.pdf
├── 삼성생명_종신보험_수수료표_최종_진짜최종.pdf
├── 삼성생명_종신보험_수수료표_202601.pdf
└── 삼성생명_종신보험_수수료표_NEW.pdf

→ 어떤 게 진짜 최신인지 모름
```

**상태 관리가 있다면:**

```
[시스템]
├── 삼성생명_종신보험_수수료표 v1.0 [DEPRECATED] 만료됨
├── 삼성생명_종신보험_수수료표 v2.0 [DEPRECATED] 만료됨
└── 삼성생명_종신보험_수수료표 v3.0 [ACTIVE] ← 이것만 유효!

→ 초록색(ACTIVE)만 보면 됨
```

### 실무 시나리오

**시나리오: 수수료표 갱신**

1. 현재 상황:
   - "삼성생명 종신보험 수수료표 v2.0" → **ACTIVE** (사용 중)

2. 새 수수료율 확정:
   - 담당자가 새 문서 생성
   - "삼성생명 종신보험 수수료표 v3.0" → **DRAFT** (작성 중)

3. 검토 완료:
   - 새 문서를 ACTIVE로 변경
   - "삼성생명 종신보험 수수료표 v3.0" → **ACTIVE** (사용 중)

4. 자동 처리:
   - 기존 문서가 자동으로 만료 처리됨
   - "삼성생명 종신보험 수수료표 v2.0" → **DEPRECATED** (만료됨)

**결과:** 어떤 설계사가 보더라도 최신 수수료표만 보게 됩니다.

## 5. 문서의 유통기한: 신선도

### 식품처럼 문서도 상한다

마트에서 우유를 살 때 유통기한을 확인하죠?

```
유통기한: 2026.02.15
```

이 날짜가 지나면 우유를 마시면 안 됩니다.

**문서도 마찬가지입니다.**

보험 시책은 보통 1개월 단위로 바뀝니다. 2월 시책을 3월에 사용하면 문제가 됩니다.

### 문서별 유통기한

문서 종류마다 "상하는 속도"가 다릅니다:

| 문서 종류 | 유통기한 | 이유 | 비유 |
|----------|---------|------|------|
| 시책, 수수료 | **30일** | 수시로 변경 | 우유 |
| 상품설명서, 스크립트 | **90일** | 분기마다 변경 | 햄 |
| 약관, 교육자료 | **365일** | 연 1회 변경 | 통조림 |

### 신선도의 세 단계

```
[신선도 계산]

경과일 = 오늘 - 마지막 업데이트일

신선(FRESH):    경과일 < 유통기한의 70%
주의(WARNING):  경과일 < 유통기한의 100%
만료(EXPIRED):  경과일 ≥ 유통기한
```

**시책 문서 예시 (유통기한 30일):**

| 경과일 | 비율 | 상태 | 색상 | 의미 |
|--------|------|------|------|------|
| 10일 | 33% | FRESH | 녹색 | 안심하고 사용 |
| 21일 | 70% | WARNING | 노란색 | 곧 만료, 갱신 준비 |
| 25일 | 83% | WARNING | 노란색 | 만료 임박! |
| 31일 | 103% | EXPIRED | 빨간색 | 사용 금지, 즉시 갱신 |

### 시각적 표현

시스템에서는 막대 그래프로 보여줍니다:

```
[시책 - 10일 경과]
████████░░░░░░░░░░░░░░░░░░░░░░ 33% 신선

[시책 - 21일 경과]
████████████████████░░░░░░░░░░ 70% 주의

[시책 - 31일 경과]
██████████████████████████████ 103% 만료!
```

### 왜 자동 계산인가?

수동으로 관리하면:
- "이거 언제 올린 거지?"
- "아직 유효한가?"
- 일일이 확인해야 함

자동으로 계산하면:
- 시스템이 매일 자동으로 계산
- 빨간색만 보면 갱신 필요한 문서 파악
- 담당자에게 알림 가능

### 신선도 리셋

문서 내용이 아직 유효하다면, "검토 완료"를 누르면 신선도가 리셋됩니다.

```
[검토 전]
시책 문서 - 25일 경과 - WARNING

[검토 완료 클릭]
시책 문서 - 0일 경과 - FRESH (오늘 기준으로 리셋)
```

## 6. 문서의 친구들: 관계

### 문서는 혼자가 아니다

영화를 볼 때 "관련 영화"를 추천받아 본 적 있으시죠?

```
"기생충"을 보셨다면:
- 같은 감독: "옥자", "설국열차"
- 비슷한 장르: "버닝", "마더"
- 후속작: (없음)
```

문서도 마찬가지입니다. 하나의 문서를 보면 관련된 다른 문서가 있습니다.

### 문서 관계의 네 가지 종류

#### 1. 부모-자식 관계 (상하 계층)

큰 문서와 세부 문서의 관계입니다.

```
[상품설명서] ← 부모 (큰 그림)
    ├── [약관] ← 자식 (세부 사항)
    ├── [특약 조항] ← 자식
    └── [청약서 양식] ← 자식
```

**비유:** 책과 챕터
- 책 = 부모
- 챕터 = 자식

#### 2. 형제 관계 (같은 레벨)

같은 수준의 문서들입니다.

```
[판매스크립트] ←→ [상담 FAQ] ←→ [반론 처리 가이드]
      ↑                ↑               ↑
   모두 "영업 지원 자료"로 같은 레벨
```

**비유:** 형제자매
- 같은 부모 밑에 있음
- 서로 동등한 관계

#### 3. 참조 관계 (참고하세요)

"이것도 보세요"라는 연결입니다.

```
[시책 문서] ──참조──► [수수료표]
     │
     └── "시책을 보려면 수수료표도 함께 보세요"
```

**비유:** 책의 각주
- "자세한 내용은 p.123 참조"

#### 4. 대체 관계 (버전 교체)

새 버전이 이전 버전을 대체합니다.

```
[수수료표 v3.0] ──대체──► [수수료표 v2.0] ──대체──► [수수료표 v1.0]
      ↑                         ↑                         ↑
   최신(ACTIVE)            만료(DEPRECATED)          만료(DEPRECATED)
```

**비유:** 스마트폰 업데이트
- iOS 17이 iOS 16을 대체
- iOS 16이 iOS 15를 대체

### 관계의 방향성

| 관계 | 방향 | 설명 |
|------|------|------|
| 부모-자식 | 양방향 | A가 B의 부모면, B는 A의 자식 |
| 형제 | 양방향 | A가 B의 형제면, B도 A의 형제 |
| 참조 | 단방향 | A가 B를 참조해도, B가 A를 참조하는 건 아님 |
| 대체 | 단방향 | A가 B를 대체해도, B가 A를 대체하는 건 아님 |

### 관계가 있으면 좋은 점

**관계가 없을 때:**
```
Q: "삼성생명 종신보험 시책 관련 자료 다 줘"
A: 시책 파일 1개만 줌
→ 수수료표, 상품설명서는 따로 찾아야 함
```

**관계가 있을 때:**
```
Q: "삼성생명 종신보험 시책 관련 자료 다 줘"
A: 시책 파일 + 연결된 수수료표 + 상품설명서 함께 제공
→ 원클릭으로 모든 관련 자료 확보
```

### 관계 시각화

시스템에서는 그래프로 보여줍니다:

```
                    ┌──────────────┐
                    │  상품설명서   │
                    │   (부모)     │
                    └──────┬───────┘
                           │
           ┌───────────────┼───────────────┐
           │               │               │
           ▼               ▼               ▼
    ┌──────────┐    ┌──────────┐    ┌──────────┐
    │   약관   │◄──►│ 판매스크립트│◄──►│  수수료표  │
    │ (자식)   │형제 │  (자식)   │형제 │  (자식)   │
    └──────────┘    └──────────┘    └─────┬────┘
                                          │참조
                                          ▼
                                   ┌──────────┐
                                   │   시책   │
                                   └──────────┘

## 7. 하나만 진짜: SSOT 원칙

### Single Source of Truth

SSOT는 "Single Source of Truth"의 약자입니다.

한국어로 하면: **"진실의 단일 출처"** 또는 **"하나만 진짜"**

### 문제 상황

SSOT가 없으면 이런 일이 생깁니다:

```
[공유폴더]
├── 삼성생명_종신보험_상품설명서.pdf (2025년 11월)
├── 삼성생명_종신보험_상품설명서_수정.pdf (2025년 12월)
├── 삼성생명_종신보험_상품설명서_최종.pdf (2026년 1월)
└── 삼성생명_종신보험_상품설명서_진짜최종.pdf (2026년 2월)

설계사 A: "나는 최종.pdf 쓸래"
설계사 B: "나는 진짜최종.pdf 쓸래"
설계사 C: "나는 가장 최근 날짜 거 쓸래"

→ 세 명이 다른 자료로 상담
→ 고객 혼란, 민원 발생
```

### SSOT 원칙

**"같은 자리에 유효한 문서는 딱 하나"**

```
[올바른 상태]
삼성생명 > 종신보험 > 상품설명서
└── v3.0 [ACTIVE] ← 이것만 유효!

v1.0 [DEPRECATED]
v2.0 [DEPRECATED]
```

### 분류 경로 = 유니크 키

분류 경로가 같으면, 유효한(ACTIVE) 문서는 하나만 있을 수 있습니다.

```
분류 경로: 삼성생명 > 종신보험 > 상품설명서

[허용]
- 이 경로에 ACTIVE 문서 1개 ✓
- 이 경로에 DEPRECATED 문서 여러 개 ✓ (과거 버전)
- 이 경로에 DRAFT 문서 1개 ✓ (작성 중인 새 버전)

[금지]
- 이 경로에 ACTIVE 문서 2개 ✗ → SSOT 위반!
```

### 시스템이 자동으로 막아줍니다

```
[시도]
사용자: "삼성생명 > 종신보험 > 상품설명서"에 새 문서를 ACTIVE로 등록할게

[시스템 체크]
시스템: 이미 이 경로에 ACTIVE 문서가 있네요?
       └── 기존: 상품설명서 v3.0 [ACTIVE]

[자동 처리]
시스템: 새 문서를 ACTIVE로 만들면, 기존 문서는 DEPRECATED로 바꿀게요.
       ├── 신규: 상품설명서 v4.0 [ACTIVE] ← 새로 유효
       └── 기존: 상품설명서 v3.0 [DEPRECATED] ← 자동 만료
```

### SSOT의 장점

| 상황 | SSOT 없음 | SSOT 있음 |
|------|----------|----------|
| "최신 자료 어디 있어?" | 모름, 찾아봐야 함 | ACTIVE 표시된 것 |
| "이거 써도 돼?" | 확신 없음 | ACTIVE면 OK |
| "왜 자료가 달라?" | 각자 다른 버전 사용 | 모두 같은 버전 |
| "누가 이전 버전 썼어?" | 찾기 어려움 | DEPRECATED 이력 확인 |

## 8. 모든 것을 연결하면

### 다섯 가지 개념의 조합

지금까지 배운 개념들을 조합하면:

```
┌─────────────────────────────────────────────────────────────┐
│                        문서 관리 시스템                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │

## 9. 실제 예시로 이해하기

### 예시 1: 새 시책 등록

**상황:** KB손해보험에서 2월 어린이보험 시책이 나왔습니다.

**단계별 처리:**

```
1. 문서 업로드
   └── 파일: KB손해보험_어린이보험_시책_202602.pdf

2. 분류 지정
   ├── 보험사: KB손해보험
   ├── 상품: 어린이보험
   └── 문서유형: 시책

3. 상태 설정
   └── 라이프사이클: ACTIVE (바로 사용)

4. 관계 설정
   └── 참조: KB손해보험 > 어린이보험 > 수수료표

5. SSOT 처리 (자동)
   └── 기존 1월 시책 → DEPRECATED

6. 완료
   ├── 새 시책: ACTIVE, 신선도 100%
   └── 구 시책: DEPRECATED
```

### 예시 2: 상품 개편 대응

**상황:** "든든 어린이보험"이 "든든 어린이보험 리뉴얼(2026-02)"로 개편되었습니다.

**처리 방법:**

```
[잘못된 방법]
기존 상품의 문서를 수정
→ 과거 계약 건 추적 불가

[올바른 방법]
1. 새 상품 등록
   └── "든든 어린이보험 리뉴얼(2026-02)"

2. 새 상품에 문서 등록
   ├── 상품설명서 v1.0 [ACTIVE]
   ├── 약관 v1.0 [ACTIVE]
   └── 수수료표 v1.0 [ACTIVE]

3. 상품 간 관계 설정
   └── 리뉴얼 상품 ──대체──► 기존 상품

4. 결과
   ├── 기존 상품: 모든 문서 DEPRECATED (과거 계약용 보관)
   └── 신규 상품: 모든 문서 ACTIVE (신규 판매용)
```

2. 이메일에서 검색 → 2개 파일 발견

## 10. 자주 묻는 질문

### Q1: 왜 Word, Excel은 안 되나요?

**A:** PDF, 마크다운, CSV만 지원합니다.

| 형식 | 지원 | 이유 |
|------|------|------|
| PDF | O | 표준 형식, 변조 어려움, 뷰어 안정적 |
| 마크다운 | O | 텍스트 기반, 버전 관리 쉬움 |
| CSV | O | 데이터 표준 형식 |
| Word | X | 형식 복잡, 뷰어 불안정, 버전 충돌 |
| Excel | X | CSV로 변환 권장 |
| PPT | X | 형식 복잡, 뷰어 불안정 |

### Q2: 문서를 잘못 분류하면 어떻게 하나요?

**A:** 언제든 수정 가능합니다.

문서 상세 화면에서 보험사, 상품, 문서유형을 변경할 수 있습니다. 단, 변경 후 같은 경로에 이미 ACTIVE 문서가 있으면 SSOT 위반 경고가 나옵니다.

### Q3: 삭제한 문서를 복구할 수 있나요?

**A:** 삭제는 복구가 어렵습니다.

삭제 대신 DEPRECATED(만료) 상태로 변경하는 것을 권장합니다. 만료된 문서는 검색에서 제외되지만, 필요시 이력 조회가 가능합니다.

### Q4: 신선도가 만료되면 자동으로 삭제되나요?

**A:** 아니요, 삭제되지 않습니다.

신선도 만료는 "갱신이 필요하다"는 경고일 뿐입니다. 문서 내용이 아직 유효하다면 "검토 완료"를 눌러 신선도를 리셋할 수 있습니다.

### Q5: 여러 상품에 걸친 문서는 어떻게 하나요?

**A:** 주 상품에 등록하고, 참조 관계를 설정합니다.

예: "삼성생명 전 상품 교육자료"
- 분류: 삼성생명 > (공통) > 교육자료
- 또는: 가장 관련 깊은 상품에 등록 + 다른 상품에 참조 연결

### Q6: 같은 문서를 여러 경로에 등록할 수 있나요?

**A:** 아니요, SSOT 원칙에 위배됩니다.

하나의 문서는 하나의 경로에만 있어야 합니다. 다른 경로에서도 접근해야 한다면, 참조 관계를 설정하세요.

### Q7: 시스템 없이도 이 원칙을 적용할 수 있나요?

**A:** 네, 하지만 수동 관리가 필요합니다.

엑셀로 문서 목록을 관리하고, 파일명에 규칙을 적용할 수 있습니다. 하지만 사람이 실수하면 체계가 무너지므로, 시스템 자동화를 권장합니다.

## 마무리

### 핵심 요약

| 개념 | 한 줄 설명 |
|------|----------|
| **분류체계** | 문서의 주소 (보험사 > 상품 > 문서유형) |
| **라이프사이클** | 문서의 상태 (작성 중 → 사용 중 → 만료) |
| **신선도** | 문서의 유통기한 (30/90/365일) |
| **관계** | 문서 간 연결 (부모-자식, 형제, 참조, 대체) |
| **SSOT** | 같은 자리에 유효 문서는 하나 |

### 이 시스템을 쓰면

1. **찾기 쉽다**: 분류체계로 정확한 위치
2. **믿을 수 있다**: ACTIVE만 보면 최신
3. **놓치지 않는다**: 신선도로 갱신 알림
4. **연결된다**: 관계로 관련 문서 함께 조회
5. **혼란 없다**: SSOT로 중복 방지
