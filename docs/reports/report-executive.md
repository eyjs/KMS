## 3. 왜 이 방안인가?

### 고민한 선택지

| 방안 | 결과 | 이유 |
|------|------|------|
| 올인원 시스템 | 기각 | 기술 역량 부재, 실패율 96% |
| 전문 인력 채용 | 기각 | 예산 부족, 채용 불확실성 |
| 전체 외주 | 기각 | 비용 과다, 기술 종속 |
| **체계 관리 + 외주 연동** | **채택** | 현실적, 성공 확률 70% |

### 핵심 통찰

```
1. 데이터 엔지니어링은 역량 밖
   → PDF 파싱, AI 처리는 외부에 위임

2. 진짜 자산은 "체계"
   → 문서가 어디에, 무엇이, 어떻게 연결되는지
   → 이것이 데이터 주권의 핵심

3. 할 수 있는 것에 집중
   → .NET + Vue = 직접 구현 가능
   → 복잡한 건 나중에 확장
```

## 부록: Q&A

**Q: 왜 AI/RAG를 직접 안 하나?**
> Python 운영 역량 없음. IT팀이 .NET만 가능. 무리하면 프로젝트 실패.

**Q: 외주가 문제 생기면?**
> API 표준화되어 있어 1일 만에 교체 가능. 데이터 주권은 우리가 보유.

**Q: 나중에 확장 가능?**
> Phase 3로 설계해둠. 예산/인력 확보 시 진행.

**Q: 왜 Word/PPT는 막나?**
> 뷰어 불안정, 형식 복잡. PDF/Markdown/CSV만 허용해서 안정성 확보.
