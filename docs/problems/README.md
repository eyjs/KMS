## 핵심 결정

### 채택된 방안: Option A (체계 관리형)

| 항목 | 결정 |
|------|------|
| 구현 범위 | 문서 체계 관리 + 뷰어 + API |
| 제외 범위 | PDF 파싱, NLP, 벡터 DB, RAG |
| 기술 스택 | .NET Core + Vue 3 + PostgreSQL |
| 개발 인력 | 1명 (직접 구현) |
| 개발 기간 | 3개월 |
| 성공 확률 | 70% |

### 왜 이 방안인가

```
1. 데이터 엔지니어링은 역량 밖
   → PDF 파싱, NLP, 벡터 DB는 Python 필요
   → IT팀 Python 운영 불가
   → 외주에 위임

2. 진짜 자산은 "체계"
   → 문서 분류, 관계, 라이프사이클
   → 이것이 데이터 주권의 핵심

3. 확장 가능하게 설계
   → Phase 3에서 선택적 확장
```
